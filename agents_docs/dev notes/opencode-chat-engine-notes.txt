## Summary
Adds OpenCode as a native Panes chat engine alongside Codex and Claude. The backend runs OpenCode through the local headless server, maps streaming events into Panes chat events, handles OpenCode permissions/questions, tracks usage/cost, exposes runtime agents/commands/MCP state, and supports remote session attach/archive/restore. The frontend exposes OpenCode in chat/onboarding, keeps Codex sandbox controls out of OpenCode threads, adds OpenCode-specific permission labels and readiness flow, groups OpenCode models by provider, exposes model variants as reasoning-effort controls, and adds a searchable OpenCode sessions panel.
## Changes
- Added the OpenCode backend engine, model discovery, health check, local server/session lifecycle, SSE event mapping, permission/question response routing, interruption, and cleanup.
- Added OpenCode engine capabilities and execution-policy normalization so OpenCode uses ask/allow/deny permission modes instead of Codex sandbox modes.
- Added OpenCode frontend support in the model/chat surfaces, attachment filtering, permission picker, onboarding engine selection, readiness install path, notification labels, and i18n resources.
- Added an OpenCode provider/model tree in the chat model picker so large OpenCode catalogs stay browsable by provider.
- Normalized OpenCode permission button responses to OpenCode-native decision payloads so permission approvals do not receive Codex-style permission payloads.
- Parsed `opencode models --verbose` model variants and sent the selected OpenCode reasoning effort as the OpenCode `variant` field.
- Added OpenCode usage/cost mapping, richer question answer handling, model attachment metadata, selected-model attachment validation, runtime agent picker, OpenCode slash command/MCP panels, and session browsing/attach lifecycle controls.
- Treated OpenCode session permission rules as create-time state: if a resumed session has different ask/allow/deny rules than Panes requests, Panes starts a fresh OpenCode session and persists the new engine session id on the next turn.
- Added tests for OpenCode backend routing/capabilities, model metadata, attachment validation, usage mapping, session metadata, frontend capability fallbacks, onboarding persistence/readiness, and OpenCode question approval action gating.

## Notes for review
Review OpenCode server lifecycle, approval/question routing, model attachment validation, runtime catalog loading, remote session attach/archive/restore, and permission-mode resume behavior closely. The frontend intentionally hides sandbox/network controls for OpenCode because OpenCode permissions are not equivalent to Panes/Codex sandboxing. Granular OpenCode permission-rule editing is intentionally deferred.
